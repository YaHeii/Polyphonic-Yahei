# AGENTS.md - 博客系统高可用微服务（单机部署版）开发规范

## 1. 架构定型
* **模式**：基于 `go-zero` 的伪分布式微服务架构。
* **通信**：单机部署，API 与 RPC 之间关闭 Etcd，使用 `Direct` 模式（配置固定本地端口）直连。
* **存储**：PostgreSQL (关系+JSONB+GIN) + Redis (缓存)。

## 2. 目录与解耦边界
* 模块间的强隔离通过独立的 RPC 服务（进程级）实现。
* 进程内部放弃基于 Interface 的传统内部分层，Logic 层可直接调用 `goctl` 生成的 Model 层，以完整保留底层自动缓存红利。
* 跨模块的数据获取，必须通过 gRPC 客户端调用对应的 RPC 服务，严禁在 A 服务中引入 B 服务的 Model。

## 3. PostgreSQL 深度应用规范
* **动态扩展**：实体多变属性必须通过 `jsonb` 字段承载，减少 DDL 变更。
* **标签设计**：标签使用数组类型 `text[]`，禁止创建传统多对多中间表。
* **检索实现**：全文检索必须使用 PgSQL 的内置文本检索与 GIN 倒排索引实现，严禁引入 Elasticsearch。

## 4. 缓存与性能防线
* 涉及高频读取（文章详情、首页列表）必须配合 go-zero 自带的缓存机制。
* 批量或可能引发大流量倾轧的逻辑，必须使用 `collection.SharedCalls` 进行包裹防击穿。

## 5. Agent 编码指令
* 增加新功能前，先确认应归属于现有哪个 RPC/API 路由，或者是否需要启动新的服务。
* 修改接口必须从修改 `.api` 或 `.proto` 文件开始，使用 `goctl` 驱动。
* 涉及修改零值字段，必须使用 `map[string]interface{}` 进行更新。

## 6.
凡是属于对第三方开源框架（如 GORM、go-zero、grpc）进行二次开发、功能缝合、或行为扩展的自定义基础设施组件，其包名必须以 x 结尾（如 gormlogx、jwtx）。此举用以明确标识该包属于非业务核心的技术外壳扩展，严禁在其中夹带任何博客系统的业务逻辑。
