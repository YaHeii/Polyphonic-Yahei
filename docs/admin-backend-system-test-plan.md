# Admin 后端系统测试方案

## 1. 目标

当前本地链路已经具备：

- `docker compose` 可启动
- `admin-api` / `blog-rpc` 可联通
- Apifox 已导入 Swagger
- 已完成登录、拿 token、健康检查、简单 GET

接下来测试目标不是“把所有接口点一遍”，而是分层验证下面四件事：

1. 服务是否可用
2. 接口契约是否稳定
3. 核心业务 CRUD 是否闭环
4. 异常路径、状态一致性、日志副作用是否合理

## 2. 测试分层

### 2.1 冒烟测试

先验证系统是否“活着”：

- `GET /admin-api/v1/ping`
- 登录成功
- 拿到 `uid` 和 `access_token`
- 用登录态访问一个受保护接口，例如：
  - `GET /admin-api/v1/user/get_user_info`
  - `GET /admin-api/v1/user/get_user_roles`
  - `GET /admin-api/v1/user/get_user_menus`
- Swagger 页面可访问
- `docker compose logs` 中没有明显 panic / connect refused / timeout

通过标准：

- 容器都在运行
- API 返回结构符合统一响应格式
- 登录后至少一个受保护接口成功

### 2.2 鉴权与协议测试

重点验证“接口边界”，不是业务本身：

- 不带 `uid`
- 不带 `authorization`
- `uid` 正确但 token 为空
- token 非法
- token 与 `uid` 不匹配
- 请求方法错误
- `Content-Type` 错误
- 请求体字段缺失
- 字段类型错误
- 分页接口传空分页参数、非法分页参数

当前仓库有两个现实约束需要注意：

- 鉴权头是 `uid` + `authorization`
- `authorization` 目前直接传原始 `access_token`
- 权限校验当前使用 `AllowAllHolder`，也就是本地环境不会真正拦权限

所以现在**可以测 token 是否生效，但不能有效测 RBAC 权限收敛**。

### 2.3 核心 CRUD 闭环测试

不要按技术模块测，按“业务最小闭环”测。推荐优先级如下。

#### A. 账号与权限基础数据

- 用户信息查询
- 角色列表
- 菜单列表
- API 列表
- 账号列表

目标：

- 确认后台基础运行依赖的数据可正常读取
- 确认 seed 数据、角色、菜单、接口基础信息正常

#### B. 内容管理

这是后台最核心的一组，建议优先测：

- 分类：新增 -> 列表 -> 修改 -> 删除
- 标签：新增 -> 列表 -> 修改 -> 删除
- 文章：新增 -> 查询详情 -> 列表 -> 修改 -> 置顶/删除状态 -> 删除
- 页面：新增 -> 列表 -> 修改 -> 删除

目标：

- 验证主业务写链路
- 验证关联字段是否正常回填
- 验证列表查询与详情查询是否一致

#### C. 媒体与扩展内容

- 相册：新增 -> 列表 -> 修改 -> 删除状态
- 图片：列表、详情、删除
- 友链：新增 -> 列表 -> 修改 -> 删除
- 说说 / 消息 / 评论：至少覆盖列表和状态流转

当前上传链路如果你暂时不测 OSS，可以先跳过真实上传，只测已有资源查询和后台管理接口。

#### D. 系统配置与日志

- 网站配置查询 / 修改
- 公告查询 / 修改
- 登录日志列表
- 操作日志列表
- 访问日志列表
- 文件日志列表

目标：

- 确认后台“可运营面板”不是只剩 CRUD 壳子
- 确认副作用日志能被写入并被查到

## 3. 每类接口的标准动作

对于每个资源类接口，统一按下面顺序测：

1. 新增
2. 查询详情
3. 查询列表
4. 修改
5. 再次查询详情
6. 删除或更新删除状态
7. 再查列表确认状态变化

这套顺序的意义：

- 能发现“写成功但查不到”
- 能发现“详情和列表字段不一致”
- 能发现“逻辑删除/物理删除行为不一致”

## 4. Apifox 里的组织方式

建议直接建 5 个测试分组，不要平铺所有接口：

1. `00-冒烟`
2. `01-鉴权`
3. `02-基础数据`
4. `03-内容管理`
5. `04-系统配置与日志`

再补两个公共能力：

- 环境变量
  - `base_url`
  - `uid`
  - `access_token`
- 登录接口后置脚本
  - 自动提取 `data.user_id`
  - 自动提取 `data.token.access_token`

这样后续跑回归时，不需要每次手工复制 token。

## 5. 数据验证不要只看 HTTP 响应

至少对“新增 / 修改 / 删除”类接口，加一层数据验证：

- 查接口返回
- 查数据库落库结果
- 查日志副作用

本地可以直接用：

- `docker compose logs admin-api`
- `docker compose logs blog-rpc`
- `docker compose exec postgres psql -U root -d blog-init`

重点看：

- 是否真正写库
- 是否只改了状态位
- 是否写入登录日志 / 操作日志

## 6. 当前阶段不值得优先做的测试

在你当前环境下，下面这些先不作为第一优先级：

- 第三方登录联调
- 邮件发送真实可达性
- OSS 真实上传
- 分布式权限一致性
- 高并发压测

原因很简单：这些都不属于“后台基础 CRUD 可联调”的最小闭环。

## 7. 推荐执行顺序

### 第一轮：把主链路打通

- 冒烟
- 登录
- 用户信息
- 菜单 / 角色 / API / 账号列表
- 分类 / 标签 / 文章 CRUD

### 第二轮：补内容外围

- 页面
- 相册 / 图片
- 友链
- 评论 / 消息 / 说说

### 第三轮：补配置和日志

- 网站配置
- 公告
- 登录日志
- 操作日志
- 访问日志

### 第四轮：补异常路径

- 缺 header
- token 错误
- 参数缺失
- 类型错误
- 非法 ID

## 8. 成功标准

本地这一阶段可以把“测试通过”定义为：

1. `admin-api` 与 `blog-rpc` 稳定运行
2. 登录态接口整体可调用
3. 至少一条核心内容链路闭环成功：
   - 分类
   - 标签
   - 文章
4. 基础后台数据可读：
   - 用户
   - 菜单
   - 角色
   - API
5. 操作后的数据库状态与接口响应一致
6. 关键异常请求能返回可预期错误，而不是 panic

## 9. 下一步建议

当你按上面流程跑完第一轮之后，下一步最有价值的是两件事：

1. 从 Apifox 手工用例里沉淀一套回归集合
2. 选最核心的 5 到 10 条链路，转成自动化测试

推荐第一批自动化目标：

- 登录
- 获取用户信息
- 分类 CRUD
- 标签 CRUD
- 文章新增 / 查询 / 修改 / 删除状态
