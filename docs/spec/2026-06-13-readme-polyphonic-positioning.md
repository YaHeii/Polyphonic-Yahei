# README 文案定位约束

状态：已确认，可直接落盘

## 1. 文档目的

本文用于收敛仓库根 `README.md` 的文案定位、信息结构与事实边界。

本次目标不是补一份通用模板，而是为 `Polyphonic-Yahei` 写一份有个人辨识度、同时能让开发者快速看懂项目形态的 README。

## 2. 定位结论

- 主标签围绕 `Polyphonic / 复调`
- 文风采用“半自嘲半硬核”
- 读者同时面向普通访客与开发者
- `个人特点` 与 `项目描述` 必须分开写，不混成一段

## 3. 结构约束

README 按以下顺序组织：

1. 标题与一句话副标题
2. `Polyphonic` 的命名解释
3. `关于我`
4. `关于这个项目`
5. 项目特点 / 技术栈
6. 目录速览
7. 快速启动
8. 生成链路说明
9. 收束式结尾

## 4. 文风要求

- 允许轻微自嘲，但不能把 README 写成段子合集
- 允许有表达欲，但不能把项目说明写成散文
- 项目硬信息要足够具体，避免“现代化、高性能、优雅”这类空词堆砌
- 优先写“这个项目为什么这样组织”，而不是泛泛列功能

## 5. 事实边界

README 只写当前仓库中已能确认的事实：

- 项目主干是 Go 后端工程
- 管理端 API 真相源位于 `service/api/admin/proto/*.api`
- Blog RPC 真相源位于 `service/rpc/blog/proto/blog/*.proto`
- 数据模型生成链路围绕 PostgreSQL 与 `goctl model`
- 本地联调主链路围绕 `docker compose`、`postgres`、`redis`、`rabbitmq`、`blog-rpc`、`admin-api`
- Swagger 由后端生成，产物位于 `service/api/admin/internal/docs/swagger.json`

以下内容不应写成“已经完成”：

- 仍处于方案或逐步推进阶段的管理后台前端建设
- 未在当前仓库事实中确认的部署平台、监控体系或线上地址

## 6. 成功标准

- 读者能在前半段理解 `Polyphonic` 的个人表达
- 读者能在后半段快速知道项目技术形态与启动方式
- 全文中文表达自然，语气诙谐但不过火
- README 中的命令、目录和模块名称与当前仓库一致
