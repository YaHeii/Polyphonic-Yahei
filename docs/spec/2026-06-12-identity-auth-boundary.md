# 身份与验证边界约束

状态：已确认约束，待实施

## 1. 文档目的

本文用于收敛 `Polyphonic-Yahei` 的账号身份、登录入口、绑定关系与验证码职责边界。

本文是后续调整 `service/api/admin/proto/admin/*.api`、`service/rpc/blog/proto/blog/account.proto`、相关 logic 与测试的约束真相源。

本文描述的是目标约束，不代表仓库当前已经全部实现。

## 2. 目标边界

### 2.1 身份源

- `email` 是唯一身份源。
- “账号是否存在”以邮箱账号是否已建立为准。
- 任何新账号都必须先通过邮箱完成注册建立。

### 2.2 绑定关系

- `username` 保留，且继续要求在注册阶段填写。
- `username` 是登录别名与站内标识，但不是独立身份源。
- `phone`、`oauth` 都是邮箱账号建立后的附属绑定入口。
- 这些绑定项都不能独立创建账号，只能挂载到一个已存在的邮箱账号。

### 2.3 登录入口

- 登录入口可以保留 `email`、`username`、`phone`、`oauth`。
- 这些入口的共同前提是：目标账号已经由邮箱注册建立完成。
- `phone` 与 `oauth` 的登录语义是“查找一个已经绑定到邮箱账号的入口”，不是“首次登录即开户”。

## 3. 核心业务规则

### 3.1 注册

- 注册只允许通过邮箱创建账号。
- 注册成功后，系统得到一个基础邮箱账号。
- 注册阶段必须同时提供 `username`，但 `username` 不是开户前提，只是注册期一并采集的登录别名。

### 3.2 用户名

- `username` 继续保留唯一性约束。
- `username` 是登录别名与站内标识，不是开户前提。
- 本轮不新增 `BindUsername`，也不把 `username` 改为后绑定流程。

### 3.3 手机号

- `phone` 只允许绑定到已有邮箱账号。
- `phone` 绑定完成后，可以作为登录入口使用。
- `phone` 未绑定时，不允许通过手机号创建账号，也不允许手机号首次登录自动开户。

### 3.4 第三方 OAuth

- `oauth` 只允许绑定到已有邮箱账号。
- `oauth` 绑定完成后，可以作为登录入口使用。
- 第三方首次登录若未绑定现有邮箱账号，必须进入绑定流程，不能自动创建新用户。

### 3.5 重置密码与身份确认

- 重置密码基于邮箱身份进行。
- “找回/确认身份”的默认主链路是邮箱。
- 手机号与第三方只作为已绑定入口，不承担主身份恢复职责。

## 4. 验证职责边界

### 4.1 API 层职责

- API 层负责所有前置验证与反滥用校验。
- API 层可以保留：
  - base64 图形验证码能力
  - 邮件验证码发送与校验
  - 手机验证码发送与校验
  - 第三方 OAuth code 换取外部身份信息
- API 层在完成验证后，再调用 RPC 执行业务动作。

### 4.2 RPC 层职责

- RPC 层只处理“已通过前置验证后的业务动作”。
- RPC 层不再承担验证码、人机校验、短信码、邮件码等验证语义。
- RPC 层请求中不应再出现：
  - `verify_code`
  - `captcha_key`
  - `captcha_code`
  - 其他同类验证字段

### 4.3 凭证与验证的区分

- 密码属于登录凭证，可继续作为账号领域输入存在于 RPC。
- 图形验证码、短信码、邮件码、OAuth 外部换码，属于前置验证或外部身份确认，不应继续直接暴露为 `AccountRpc` 输入字段。

## 5. 接口约束

### 5.1 API 层

- API 层可以继续暴露多入口登录接口。
- 但每个入口都必须在 API 边界先完成对应验证，再调用 RPC。
- 用户绑定类接口若需要图形验证码、短信验证码或邮件验证码，也应在 API 层完成校验。

### 5.2 RPC 层

- `AccountRpc` 应表达账号领域动作，而不是验证码流程。
- `Register`、`ResetPassword`、`EmailLogin`、`PhoneLogin`、`BindUserEmail`、`BindUserPhone` 等请求应收敛为纯业务字段。
- `GenerateCaptchaCode`、`SendEmailVerifyCode`、`SendPhoneVerifyCode` 这类能力不应继续作为长期的账号领域 RPC 真相源。

### 5.3 第三方登录输入

- API 层负责用 `platform + oauth code` 换取稳定外部身份。
- RPC 层应接收“已确认过的平台身份信息”或“已绑定关系所需信息”，而不是直接承担外部授权码换取逻辑。

## 6. 对当前项目的约束结论

### 6.1 必须保留

- 邮箱注册
- 邮箱密码登录
- 用户名登录入口
- 手机号登录入口
- 第三方登录入口
- 邮箱、手机号、第三方的绑定能力
- 注册阶段填写 `username`

### 6.2 必须收口

- 邮箱是唯一开户前提。
- `phone` 改为后绑定。
- `oauth` 改为后绑定。
- 第三方首次登录不再自动注册。
- RPC 中不再保留验证字段。

### 6.3 明确不允许

- 手机号直接注册新账号
- 第三方首次登录自动创建新账号
- 未建立邮箱账号前直接绑定 `phone`、`oauth`
- 在 `AccountRpc` 中继续扩散新的验证码语义字段

## 7. 当前实现与目标之间的主要差距

- 当前注册虽然要求 `username`，但 RPC 仍混入邮箱验证码语义。
- 当前 `PhoneLogin` 仍直接以手机号验证码作为 RPC 输入。
- 当前 `BindUserEmail`、`BindUserPhone`、`Register`、`ResetPassword` 等请求仍携带 `verify_code`。
- 当前 `EmailLogin` 仍把 `captcha_key/captcha_code` 放进 RPC。
- 当前第三方登录在未找到绑定关系时会自动注册新用户，与“邮箱先开户”冲突。
- 当前账号领域下仍承载 `GenerateCaptchaCode`、`SendEmailVerifyCode`、`SendPhoneVerifyCode` 等验证流程能力。

## 8. 后续实施时必须遵守的原则

- 先改真相源，再改生成产物。
- API 契约与 RPC 契约必须分别表达自己的职责，不混用字段语义。
- 若某个登录/绑定流程仍需要验证码，也应优先把验证放在 API 边界完成，而不是继续把验证字段透传到 RPC。
- 若短期需要兼容旧接口，应在实施文档中明确兼容窗口；但长期目标不变，即 RPC 去验证语义、邮箱为唯一开户前提。
